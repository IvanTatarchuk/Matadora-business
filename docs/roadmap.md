# Roadmap — пріоритети розвитку Matadora

Цей документ синтезує дослідницькі звіти (`docs/research/*.md`) в один порядок дій.
Агент-Розробник орієнтується на секцію "Технічні завдання" зверху вниз, по одному
завданню за раз.

**Ревізія:** 2026-07-03

## 🥇 Пріоритет №0 — швидкий кешфлоу (терміново)

**Переналаштувати `/kosztorys` під нову аудиторію: архітектори, управителі
нерухомості, дизайнери інтер'єру** — не тільки генеральні підрядники.

Обґрунтування (дослідження "willingness-to-pay" + "DFY vs SaaS"): рушій
"PDF→кошторис" уже готовий і оплачений через Stripe. Найшвидший шлях до
першого доходу — не будувати нове, а продавати той самий інструмент іншій
аудиторії напряму (DFY-послуга за документ), а не чекати самообслуговування.

Технічно потрібно (для агента-Розробника):
- Оновити копірайтинг на `/kosztorys` — прибрати мову, орієнтовану виключно
  на "generalny wykonawca", додати мову для архітекторів/управителів
  нерухомості/дизайнерів (напр. "оцінка вартості ремонту для клієнта" замість
  тільки "kosztorys budowlany")
- Перевірити, чи приклади/дефолтні KNR-позиції залишаються релевантними для
  цієї аудиторії (вони достатньо загальні — ремонт/оздоблення, мають підійти)
- НЕ чіпати сам розрахунковий рушій (`computeOfferTotals`, KNR-каталог,
  Stripe-інтеграцію) — тільки позиціонування/копірайт і, за потреби, нову
  секцію на лендінгу "для кого цей інструмент"
- Малий, безпечний обсяг змін — тільки текст/UI, без зміни схеми БД

Продаж і клієнтський аутріч — це дія власника, не агента (див. RECOVERY.md
принципи меж автоматизації).

## Технічні завдання (для агента-Розробника, у порядку пріоритету)

1. **Валідація кошторису перед відправкою** — попередження про нульову ціну,
   ставку-викид, відсутню назву клієнта. Мала складність, без зміни схеми.
   Джерело: `openconstructionerp-analysis.md` §2.5.

2. **EVM S-curve графік** — дані вже є в `evm_snapshots`
   (`bac`/`pv`/`ev`/`ac`/`cpi`/`spi`), просто немає жодної бібліотеки графіків
   у всьому коді. Чиста фронтенд-задача на `evm-client.tsx`.
   Джерело: `platform-wide-feature-analysis.md` §4.1.

3. **Картка "готівка зараз + прогноз на 4-8 тижнів"** — агрегація
   `cashflow_entries` + `invoices` + `retention_payments` в одну картку на
   `/dashboard/contractor/finanse`.
   Джерело: `platform-wide-feature-analysis.md` §4.2.

4. **Бейдж кваліфікації підрядника біля ставки** — прострочена страховка/
   ліцензія видно одразу в списку ставок, дані вже є, тільки читання.
   Джерело: `platform-wide-feature-analysis.md` §1.3.

5. **Матриця порівняння ставок side-by-side** — замість двох окремо
   відсортованих списків.
   Джерело: `platform-wide-feature-analysis.md` §1.1.

6. **Історія оцінок підрядника по проєктах** — нова таблиця
   `subcontractor_reviews` замість однієї перезаписуваної `rating`.
   Джерело: `platform-wide-feature-analysis.md` §3.1.

## Стратегічний трек — НЕ для агентів, потребує людини

1. **Партнерство з факторинговою компанією** (Bibby Financial Services,
   INDOS, PKO Faktoring) — реферальна комісія 10-20%. Дзвінок/лист власника.
2. **DFY-продажі `/kosztorys` новій аудиторії** — перші 5-10 повідомлень
   архітекторам/управителям нерухомості з наявної мережі контактів.
3. **Розмова з księgowy** про легалізацію діяльності — передумова для
   факторингу й будь-якого зовнішнього фінансування.

## Свідомо відкладено

- KSeF live-submission API — власник прямо вирішив не інвестувати сюди понад
  обов'язковий мінімум. **Не пропонувати як пріоритет.**
- Offline-режим для роботи на об'єкті — велика робота, немає ще запиту.
- Суперечки/претензії, гарантії/страхування — дослідження прямо каже "докази
  слабкі, не пріоритет".
- Маркетплейс матеріалів, cost-index як окремий продукт — потребують
  масштабу транзакцій, якого ще немає.
- Загальний "AI wrapper" SaaS з нуля — дослідження показало це як
  найпереоціненіший, найменш надійний шлях до доходу.

## Що ще не відомо

- Поточна кількість реальних платних клієнтів і unit-економіка pay-per-use
  kosztorys — внутрішні дані, не публічна інформація.
