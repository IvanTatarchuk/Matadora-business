# Процес релізів

Мета: щоб кожне помітне оновлення платформи було окремою версією з
чіткими нотатками, а не тихим потоком комітів у production.

## Версіонування

Semantic Versioning (`MAJOR.MINOR.PATCH`), напр. `1.2.0`:
- **PATCH** (`1.0.1`) — виправлення багів, дрібні правки тексту/UI, без нових функцій.
- **MINOR** (`1.1.0`) — нова функція, зворотно сумісна (напр. новий модуль, нова платна AI-функція).
- **MAJOR** (`2.0.0`) — ламаючі зміни (зміна структури даних, видалення функції, несумісний API).

## Гілки

- **`main`** — робоча гілка. Сюди агенти комітять і пушать (включно з
  автоматичним `auto-backup`). Vercel деплоїть `main` як **Preview**
  (staging), не production.
- **`production`** — те, що реально бачать користувачі на matadora.business.
  Просувається вперед лише свідомим релізом (злиттям з `main`), ніколи
  напряму.

> **Одноразове налаштування (дія власника, не агента):** у Vercel-проєкті
> `buildmate-app` → Settings → Git → Production Branch змінити з `main` на
> `production`. Це єдина ручна дія, потрібна щоб цей процес запрацював —
> Vercel CLI не дає змінити цю настройку програмно (перевірено: `vercel
> project`/`vercel git` не мають такої команди), тому це так само, як
> застосування міграцій — залишається за власником.
>
> До того, як це зроблено, `main` = production (як зараз), і секція
> "Реліз" нижче зводиться до кроків 1-4 (без злиття в `production`).

## Як зробити реліз

1. **Оновити `CHANGELOG.md`** — перенести вміст `[Unreleased]` у нову
   секцію `[X.Y.Z] — YYYY-MM-DD`, розписати по підрозділах (Додано /
   Виправлено / Змінено / Видалено). Залишити `[Unreleased]` порожньою
   зверху для наступних змін.
2. **Підняти версію в `package.json`** (`"version": "X.Y.Z"`).
3. **Закомітити:** `chore(release): vX.Y.Z`.
4. **Тег + GitHub Release:**
   ```bash
   git tag -a vX.Y.Z -m "vX.Y.Z"
   git push origin vX.Y.Z
   gh release create vX.Y.Z --title "vX.Y.Z" --notes-file <(sed -n '/## \[X.Y.Z\]/,/## \[/p' CHANGELOG.md | sed '$d')
   ```
   (або вручну скопіювати відповідну секцію CHANGELOG.md у нотатки релізу).
5. **Просунути в production** (після того, як налаштовано Production
   Branch = `production`, див. вище):
   ```bash
   git checkout production
   git merge main --ff-only
   git push origin production
   git checkout main
   ```
   Vercel автоматично задеплоїть `production` на matadora.business.
6. Перевірити продакшн (`curl`/живий тест ключових маршрутів) після
   деплою, як робиться зараз.

## Що показувати користувачам

- `/changelog` — публічна сторінка, польською, коротким, орієнтованим на
  користувача списком (не технічний CHANGELOG.md один в один — перекласти
  й спростити формулювання).
- Версія платформи видима в футері сайдбару дашборду.
