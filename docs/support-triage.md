# Тріаж тікетів підтримки

Звідки беруться тікети: віджет "Zgłoś problem" (плаваюча кнопка, видима на
всіх сторінках дашборду для всіх трьох ролей) → таблиця `support_tickets`
→ email-сповіщення власнику (`SUPPORT_ADMIN_EMAILS`, за замовчуванням
`itatarchuk1202@gmail.com,vanbud.felix@gmail.com`).

**Жорстке правило: тріаж НІКОЛИ не деплоїть і не мержить у `main`/
`production` самостійно.** Максимум, що робить тріаж — комітить фікс у
нову гілку і пушить її. Далі PR відкриває й мержить власник вручну.
Немає `gh` CLI/токена в цьому середовищі — тому PR не створюється
програмно; замість цього тікет отримує готове посилання
`https://github.com/IvanTatarchuk/Matadora-business/compare/main...<гілка>?expand=1`,
яке одразу відкриває форму створення PR на GitHub одним кліком.

## Коли запускати

Власник просить ("протріаж тікети" / "перевір скриньку підтримки") —
це свідома дія людини, не крон і не тригер на кожен новий тікет.

## Процес (для агента, що виконує тріаж)

1. Отримати відкриті тікети:
   ```sql
   select * from support_tickets where status = 'open' order by created_at;
   ```
   (через `createAdminClient()` / службовий ключ — RLS дозволяє читати лише
   власні тікети repoter'у, тріаж працює з обходом RLS, як і решта
   адмін-операцій у цьому проєкті.)

2. Для кожного тікета:
   - Прочитати `message` і `page_url`.
   - Знайти відповідний код (за `page_url` — шлях зазвичай напряму
     відповідає `src/app/...`).
   - Визначити: це **конкретний, локалізований баг у коді**, чи щось інше
     (питання користувача, запит фічі, проблема з даними, неоднозначний
     опис, потребує продуктового рішення)?

3. **Якщо НЕ чіткий баг у коді** (найчастіший випадок):
   - НЕ чіпати код.
   - `update support_tickets set status='triaging', triage_notes='<пояснення власнику, що це і чому не є автофіксом>' where id=...`
   - Залишити власнику вирішувати вручну.

4. **Якщо чіткий, локалізований баг у коді** (малий і безпечний обсяг —
   одна сторінка/компонент/функція, без міграцій, без зміни платіжної
   логіки, без зміни RLS/прав доступу):
   - `git checkout -b fix/ticket-<коротких-8-символів-id>-<slug>` від
     актуального `main`.
   - Внести фікс.
   - `npm run typecheck && npm run lint && npm run build` — усе має пройти.
   - Живий тест, якщо є практична можливість (curl/браузер), як і для
     будь-якої іншої зміни в цьому проєкті.
   - Закомітити з описовим повідомленням, що посилається на ID тікета.
   - `git push origin fix/ticket-<...>`.
   - `git checkout main` (повернутись, не залишатись у гілці фіксу).
   - Оновити тікет:
     ```sql
     update support_tickets
     set status='fix_drafted',
         fix_branch='fix/ticket-...',
         fix_summary='<що і чому змінено, включно з compare-посиланням на GitHub>'
     where id=...;
     ```

5. **Якщо баг потенційно зачіпає щось із заборонених для агента зон**
   (`supabase/migrations/`, зміна платіжної/Stripe-логіки, RLS-політики,
   зміна цін) — НЕ намагатися фіксити автоматично, навіть якщо технічно
   очевидно що не так. `status='triaging'`, `triage_notes` з чітким описом
   і поясненням, чому потрібне ручне рішення власника.

## Що бачить власник

`/dashboard/support-inbox` (видимо в сайдбарі лише для email з
`SUPPORT_ADMIN_EMAILS`) — список тікетів, статус, нотатки тріажу, гілка
фіксу (якщо є), можливість відповісти користувачу текстом (лист
надсилається автоматично через Resend) і вручну змінити статус.
